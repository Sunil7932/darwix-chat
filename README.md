# Darwix AI — Chat Interface

A production-grade, accessible, high-performance chat interface built for the
Darwix AI Frontend Developer assessment.

It implements the full message lifecycle (optimistic send → delivery → retry on
failure → **streamed bot reply**), session persistence, a typing indicator, and a
**virtualised transcript** that stays smooth with thousands of messages — all
wrapped in a responsive, keyboard-navigable, screen-reader-friendly UI.

Replies are generated by **Google Gemini** through a small **server-side proxy**
so the API key never reaches the browser, and stream back token by token. With no
key configured the app still runs end-to-end against an offline fallback bot, so
it works out of the box.

> **Tech stack:** React 19 · TypeScript (strict) · Vite 6 · Tailwind CSS v4 ·
> TanStack Virtual · lucide-react · Google Gemini (server-side proxy)

## Quick Start

```bash
# Requirements: Node 18+ (developed on Node 22)
npm install
npm run dev          # start the dev server → http://localhost:5173
```

### Enabling real AI replies (optional, free)

The app runs immediately with an offline fallback bot. To get **real, streamed
AI answers for free**, add a Google Gemini key:

1. Get a **free** API key (no credit card) at **https://aistudio.google.com/apikey**.
2. Configure it:

```bash
cp .env.example .env
# then edit .env and set GEMINI_API_KEY=...
npm run dev
```

`GEMINI_API_KEY` is read **only on the server** (the `/api/chat` proxy) and is
never bundled into client code. Optionally set `GEMINI_MODEL` (defaults to
`gemini-2.5-flash`).

### Available Scripts

```bash
npm run dev          # start the dev server with hot reload
npm run build        # type-check (tsc -b) + production build to /dist
npm run preview      # serve the production build locally
npm run lint         # ESLint
npm run typecheck    # type-check only
```

### Try these things

- **Send a message** — type and press **Enter** (use **Shift + Enter** for a new line).
  With a key set, the model's reply streams in token by token.
- **Watch retry** — if a request to the model fails (network/upstream error), the
  message shows *Not delivered → Retry*.
- **Load the demo** — open the **⋮ menu → Load demo conversation** to drop in
  **1,000 messages** and watch virtualisation keep scrolling buttery-smooth.
- **Scroll up** — a *Jump to latest* pill appears with an unread count; new
  messages won't yank you back down while you read history.
- **Reload the page** — your conversation is restored from `localStorage`.

### Design Principles

- **The API key is server-side only.** The browser talks to `/api/chat`; the key
  lives in the proxy's environment and is never bundled into client code.
- **Separation of concerns** — pure logic in `lib/`, stateful orchestration in
  `hooks/`, presentation in `components/`. The UI never talks to storage directly.
- **The transcript is virtualised, but accessibility isn't sacrificed** — because
  off-screen rows aren't in the DOM, announcements are mirrored into dedicated
  ARIA live regions instead of relying on the list itself.
- **Async work is cancellable** — all fetches/timers are tied to an
  `AbortController`, so clearing the chat or unmounting cleanly cancels in-flight
  streaming requests without leaking state.
- **State updaters stay pure** — side effects are never run inside a `setState`
  updater, which React StrictMode intentionally double-invokes.

## Security

Defensive defaults throughout (OWASP-aligned secure-coding practices):

| Concern | Mitigation |
|---|---|
| **API key exposure** | Key read only by server proxy from `process.env`; sent via header, never URL; never bundled into client code |
| **Input validation (server)** | POST-only, 100 KB body cap, validates/clamps every message field (role, content type, length, count) |
| **Input validation (client)** | 4,000 char limit, trimmed whitespace, type-safe message schema |
| **XSS prevention** | All content rendered as text (React escaping); no `dangerouslySetInnerHTML`; no `eval` |
| **Storage safety** | Data from `localStorage` treated as untrusted — every field validated, schema version checked, sizes clamped |
| **Error information** | Generic error messages only; no stack traces, provider internals, or PII exposed to client |
| **Dynamic execution** | No `eval`, no dynamic code execution, no unsafe deserialization |
| **Secret hygiene** | `.env` gitignored, `.env.example` provided with empty values |

## Assumptions

- **Replies come from Google Gemini via a server-side proxy.** The browser never
  holds the key — it POSTs the conversation to `/api/chat`, which streams the
  model's reply back. Locally this runs as Vite middleware; in production it's a
  serverless function (`api/chat.ts`). Both share `server/chatHandler.ts`, so the
  provider can be swapped in one file.
- **It runs without a key.** With no `GEMINI_API_KEY`, the client falls back to
  an offline canned bot, so the full UX (typing, persistence, retry) still works.
- **Persistence is per-browser** via `localStorage` (the brief asked to *simulate*
  session persistence). It's debounced and capped at 5,000 messages.
- **A single conversation** is modelled (no multi-thread/sidebar), keeping focus
  on the depth of the chat experience the brief describes.
- **Messages render as plain text.** Model output is shown verbatim and safely as
  text. Rich Markdown rendering would be a natural next step.

## Deployment

The frontend is a static Vite build (`npm run build` → `dist/`). A `vercel.json`
is included for zero-config deployment to **Vercel**: it sets the build command,
output directory, and SPA rewrites. The `api/chat.ts` function runs as a
serverless function alongside the static build.

```bash
# Deploy to Vercel (install the Vercel CLI first: npm i -g vercel)
vercel
```

Set `GEMINI_API_KEY` (and optionally `GEMINI_MODEL`) as an environment variable
in the host's dashboard — never commit it. Without the variable, the deployed
app runs in offline-fallback mode.

## Browser Support

Modern evergreen browsers (Chrome, Edge, Firefox, Safari). Uses `crypto.randomUUID`,
`AbortController`, streaming `fetch`, `Intl`, and `dvh` units — all widely
supported — with a UUID fallback for non-secure contexts.
